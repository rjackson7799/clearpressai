# ClearPress AI

A multi-tenant SaaS tool for Japanese pharmaceutical PR firms. v1 is a single-user-class internal tool that generates compliance-aware press releases (and five other content types) in a *specific client's* brand voice, runs them through 薬機法 / PMDA compliance checks, and delivers them to clients by email with magic-link feedback.

### Phase 2 deviations from the plan (worth knowing)

- **shadcn `form` component is not in the `radix-nova` registry.** `shadcn@latest add form` silently no-ops on this preset. The canonical shadcn form.tsx was written by hand, adapted to the radix-nova convention (`radix-ui` barrel imports + `data-slot` attributes). `FormMessage` was also modified to prefer caller-supplied children over the auto-rendered error string so i18n-translated error messages can flow through.
- **shadcn `sonner` component imports `next-themes`** by default. Since this is Vite/React (not Next), and the app has no theme switcher yet, the `useTheme()` call was replaced with a hardcoded `theme="system"`.
- **pdfjs-dist v5 is jsdom-incompatible at module load** (touches `DOMMatrix` via canvas.js even when no PDF is processed). The fix: `pdfjs` and its worker setup are dynamic-imported inside the PDF branch of `extractTextFromFile`, so unit tests for TXT/DOCX paths never load it. Production behavior is unchanged.
- **Anthropic SDK pin floats at `^0.95.2` in `supabase/functions/extract-voice/deno.json`** (latest at deploy time). The plan didn't pin a version; Deno needed an actual recent one to resolve.
- **`zod` v4 was installed (`^4.4.3`), not v3.** API differs slightly — `.default()` creates an input/output type split that breaks RHF type inference, so defaults are kept in the form's `defaultValues` only, not on the schema fields.
- **Anthropic API secret managed via the Supabase dashboard** (not CLI). The deploy step ran `npx supabase functions deploy extract-voice` only — the user set `ANTHROPIC_API_KEY` directly in the dashboard. Equivalent endpoint.
- **Smoke test ran against the live Anthropic API on real AZ samples** (5 .txt from the prototype corpus, 21,588 input / 884 output tokens, 20.4s wall). The test client was deleted after; storage bucket has no orphans because the smoke flow seeded `storage_path` strings without uploading bytes.
- **Eslint config now exempts `src/components/ui/**` from `react-refresh/only-export-components`** (shadcn convention exports variants/hooks alongside components) and globally ignores `prototypes/` + `.tmp-*`.

- **Original plan claimed "zero schema changes"; audit corrected this.** Migration 0004 was required and shipped: `content_items.content_sub_type` (the user's brief-time sub-type choice; default `'auto'`), the `project_summary` view (which the ProjectsListPage queries — it existed only in TSD docs, not the actual DB), and the `regenerate_variant` PL/pgSQL function (atomic clear-findings-and-upsert; H7 audit fix replacing a JS-side delete-then-insert with a transaction).

The next `supabase db push --linked` will attempt to re-apply 0004 and fail on the existing column. Mitigation when convenient: `npx supabase migration repair --status applied 0004 --linked` (writes one row; runs no SQL).
- **`supabase gen types typescript --linked` output needed manual cleanup.** Two stray lines leaked into the type file: a `Initialising login role...` stderr message (from running with `2>&1`) and a `<claude-code-hint v="1" ... />` plugin tag appended at end-of-file. - **Deterministic compliance layer added** beyond the plan's original LLM-only design (D9 audit fix). `compliance-check` runs both passes; findings carry `[deterministic]` or `[LLM]` suffix on `regulation_reference` so Phase 4 audit-event backfill can distinguish source. ### Phase 4 deviations from the plan (worth knowing)

- **Migrations 0005 and 0006 were applied via the Supabase dashboard SQL editor, not `supabase db push --linked`** (same path as 0004). Cloud `supabase_migrations.schema_migrations` has no rows for 0004, 0005, or 0006. Before the next `db push`, run `npx supabase migration repair --status applied 0004 --linked && --status applied 0005 --linked && --status applied 0006 --linked` (or three sequential repair calls).
- **`_canonical.ts` lives in `supabase/functions/_shared/canonical.ts`, not `sign-audit-report/_canonical.ts`** as the plan T7 file list specified. Moved to `_shared/` for consistency with the existing `_shared/{cors,errors,auth,audit-events}.ts` helpers; `verify-audit-signature` imports `../_shared/canonical.ts` instead of reaching across function folders.
- **Snapshot is built in PL/pgSQL (`_build_audit_snapshot` RPC), not Deno.** Plan T7 step 2 said the sign Edge Function should load variants + findings and build the snapshot itself. That created a timestamp-format-mismatch risk between JS-built and SQL-built snapshots when `finalize_audit_report` later compares them via jsonb equality. Moving the build into PL/pgSQL eliminates the risk — both sign-time and finalize-time snapshots come from the same SQL function.
- **`canonical_payload.signed_at` is JS-computed; `audit_signatures.signed_at` column uses PG `now()` default.** They can differ by milliseconds. Verification consumes `canonical_payload`, so the HMAC stays consistent; UI should display the column value as the authoritative DB record (or canonical_payload's, but pick one).
- **`useApproveVariant` unapprove path falls back to a direct table write** (no `unapprove_variant` RPC in v1; carry-forward to v2 with audit event). Approve path uses the RPC. Same shape: `useReopenFinding` is a direct write (no `reopen_finding` RPC in v1). Both have been added to the "v2 carry-forward" list and are exempt from the I1 grep guard because they touch `content_variants` / `compliance_findings`, not `audit_trail_events` / `audit_signatures`.
- **Locale namespace not added under `audit`.** AuditReportPage and the audit components use inline `BilingualLabel` for every bilingual string (matching the rest of the Phase 3 UI). A namespace can be added later if something needs programmatic `t()` access.
- **`apply_fix` body_html cast.** The Supabase type generator marks RPC args as required, but `p_new_body_html` is `text` in SQL and accepts NULL. The hook double-casts (`null as unknown as string`) so existing nulls in `content_variants.body_html` aren't coerced to `''` on fix-apply. Comment in `useApplyFix.ts` explains.
- **`apply_fix` rejects only `'fixed'`, not `'fixed'` or `'acknowledged'`.** Earlier draft had strict rejection of any non-`'unresolved'`; the approved Q4 resolution allows `acknowledged → fixed` upgrades. The RPC records `prior_status` in the `fix_applied` event details so the trail captures the upgrade.
- **`react-hooks/set-state-in-effect` again.** The reset-on-reopen pattern in `SignAuditDialog` / `RequestRevisionDialog` was rewritten from a `useEffect` to a wrapped `onOpenChange` so eslint stays happy.
- **Live audit events have no idempotence guard.** Backfilled rows (migration 0006) use `details.source_table/source_id/source_event` keyed by the partial index `idx_audit_trail_backfill_key`. Live rows have no equivalent — a retry after a partial failure (variant insert succeeded, audit insert failed) will write a duplicate event on retry. Acceptable for v1; carry-forward to v2 to move the audit insert INTO the RPC for atomicity.
- **Smoke-time hotfix (migration 0007):** `finalize_audit_report` gate I3 #5 originally used `max(compliance_findings.created_at)` as the freshness signal, which falsely flagged clean compliance runs (zero findings) as stale because `max()` returns NULL and `COALESCE`-to-epoch defeated the comparison. 0007 swaps gate 5 to use `max(audit_trail_events.occurred_at)` for `compliance_checked`/`compliance_rechecked` events keyed on `details.variant_id`. Gate 6 was also widened to accept either event type — T4 emits `compliance_rechecked` when prior findings existed, and a variant with only that event would have failed the original gate 6 query. Both gate fixes are in the single `CREATE OR REPLACE FUNCTION` body of 0007; signature unchanged so no client-side type regeneration needed.
- **Smoke-time hotfix (`FunctionsHttpError` unwrap):** `useSignAuditReport` and `useVerifyAuditSignature` originally surfaced supabase-js's generic "Edge Function returned a non-2xx status code" message when a P0004 gate failed, instead of the actual error code. Both hooks now catch `FunctionsHttpError` and read `error.context.json()` to extract the real `error.message` (e.g., `compliance_stale`, `unresolved_blockers_exist`) so the `SignAuditDialog` P0004 → localized-message map can do its job.

### Phase 4 carry-forward to v2 (out of scope for v1)

- **Dedicated `audit_writer` DB role.** Currently `firm_users_full_access` lets any authenticated session INSERT into `audit_trail_events` / `audit_signatures` directly. The I1 grep guard enforces no client code does so, but the DB doesn't. v2 should provision a separate role with `revoke insert on audit_* from authenticated`.
- **Trigger-level append-only enforcement on `audit_trail_events`** (TSD §13.4). Deny UPDATE/DELETE at the row level.
- **Atomic audit inserts inside Edge-Function-originated RPCs.** Today `generate-variants` calls `regenerate_variant` then writes the audit event as a follow-up insert; a retry after partial failure produces duplicate events. v2 should move the audit insert into the RPC so domain + audit commit together.
- **HMAC → Ed25519 per-firm keypairs** once multi-firm tenancy lands.
- **`unapprove_variant` + `reopen_finding` RPCs** with their own audit events (currently direct table writes; not audit-bearing in v1).

## Conventions

These are project conventions worth knowing before touching code:

- **No invented spec ambiguities.** PRD and TSD have known unresolved items (tracked in the plan file). Don't paper over them with assumptions; surface them for the user to decide.
- **Bilingual labels.** Per PRD §6.3, UI labels render Japanese primary + English secondary co-display, NOT a single-language toggle. Body content stays single-language.
- **Comments.** Default to no comments. Only add when WHY is non-obvious (a constraint, an invariant, a workaround). Don't narrate WHAT well-named code already says.
- **Don't add features beyond the task.** Bug fixes don't need cleanup; one-shots don't need helpers. No premature abstractions.
- **Audit trail must record exact model version strings**, not friendly names. PRD §5.3 + TSD §10 explicitly call this out as a regulatory requirement.
- **Sample documents are copyrighted.** Press releases from real Japanese pharma companies (AZ, Chugai, etc.) live in `prototypes/brand-voice-extraction/samples/<slug>/*.txt`. The `.gitignore` blocks them; only the manifests (URL lists) are committed. Don't commit `.txt` sample bodies.
